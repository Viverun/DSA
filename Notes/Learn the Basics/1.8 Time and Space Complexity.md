### Why Time Complexity Isn't About Actual Runtime

It's tempting to think "time complexity" means literally timing how long a piece of code takes to run. It doesn't — and this distinction matters. If you ran the exact same Python script on an old laptop versus a powerful modern workstation, you'd get two different runtimes for identical code. Actual runtime depends on hardware, background processes, even which Python interpreter version you're using. None of that tells you anything meaningful about whether your _logic_ is efficient.

**Time complexity instead measures how the number of operations a piece of code performs grows as the input size grows** — completely independent of the machine running it. This is what lets you compare two different approaches to the same problem on equal footing, and it's exactly what an interviewer is evaluating when they ask about your solution's efficiency.

---

### Representing Complexity: Big O Notation

We express this growth rate using **Big O notation**, written as `O(...)`, where the expression inside describes how operations scale relative to input size (conventionally called `n` or `N`).

Let's count actual steps to see where this comes from:

```python
for i in range(1, 6):
    print("Raj")
```

Walking through what actually happens here:

1. `i` is initialized to 1 (part of `range` setup)
2. Check the loop condition (is there a next value? — conceptually, "is `i` still within range")
3. Run the print statement
4. Move to the next value of `i`
5. Check the condition again
6. Print again... and so on

The loop body effectively runs 5 times, and each pass does roughly a constant handful of operations (check, print, advance). So total steps ≈ 5 × (some constant number of operations per iteration). If we generalize the `5` to `n`, the total step count becomes proportional to `3n` (or whatever constant multiplier your step-counting lands on) — and in Big O notation, we'd call this **O(n)**.

Manually counting steps like this doesn't scale, though — real code has nested loops, conditionals, and function calls layered together, and no one counts every individual operation by hand. Instead, we follow a small set of rules that let us jump straight to the meaningful growth rate.

---

### The Three Rules for Calculating Time Complexity

**Rule 1: Always consider the worst case.** **Rule 2: Drop constant terms.** **Rule 3: Drop lower-order terms.**

Let's go through each carefully.

---

#### Rule 1: Worst Case, Best Case, Average Case

Before applying this rule, it helps to define all three terms clearly using a concrete example — the grading function from section 1.3:

```python
def grade(marks):
    if marks < 25:
        return "F"
    elif marks <= 44:
        return "E"
    elif marks <= 49:
        return "D"
    elif marks <= 59:
        return "C"
    elif marks <= 69:
        return "B"
    else:
        return "A"
```

- **Best case:** the scenario where the code does the _least_ work. If `marks = 10`, only the very first condition (`marks < 25`) is checked before returning `"F"` — just one comparison.
- **Worst case:** the scenario where the code does the _most_ work. If `marks = 70`, every single condition in the chain gets checked (all come back false) before finally falling through to the `else` — the maximum number of comparisons this function can ever perform.
- **Average case:** somewhere between the two, accounting for a realistic distribution of inputs.

**We always analyze time complexity using the worst case.** The reasoning is practical: you want to know the absolute upper bound your code could take, so you can be confident it'll perform acceptably no matter what input actually shows up — that's the scenario that actually matters for judging robustness.

---

#### Rule 2: Drop Constant Terms

Suppose precise step-counting gives you a time complexity of `O(3n + 1)`:

```python
def process(n):
    x = 2                    # 1 constant-time step, regardless of n
    for i in range(n):
        # some O(1) work happens here, 3 operations per iteration
        pass
    return x
```

The `+1` from the single `x = 2` line becomes utterly insignificant once `n` grows large — whether `n` is 100 or 10,000,000, that one extra step barely registers next to the loop's contribution. So we drop it, and simplify `O(3n + 1)` down to `O(3n)`.

#### Rule 3: Drop Lower-Order Terms (and Constant Coefficients)

Now suppose you've calculated a complexity of `O(4n³ + 3n² + 8)`. First, Rule 2 drops the `+8`, leaving `O(4n³ + 3n²)`. But consider what happens as `n` grows very large — say `n = 10⁵`: the `n³` term becomes `4 × 10¹⁵`, while the `n²` term is only `3 × 10¹⁰`. That second term is many orders of magnitude smaller and contributes almost nothing to the total by comparison. **So we drop lower-order terms entirely**, leaving `O(4n³)`.

We go one step further, too: even the constant coefficient `4` gets dropped, since it doesn't change the fundamental _rate_ of growth. Final answer: **O(n³)**.

This is why you'll almost never see time complexities written as `O(4n³ + 3n² + 8)` in practice — always simplify down to the single dominant term, with no coefficient.

---

### A Brief Note on Other Notations

Big O describes an **upper bound** (worst case). Two other notations exist in formal complexity theory: **Theta (θ)**, which describes a _tight_ bound (a scenario's behavior is bounded both above and below by the same rate), and **Omega (Ω)**, which describes a _lower_ bound (best case). For interview and practical DSA purposes, Big O is what you'll use essentially all the time — the other two are worth recognizing by name, but you won't need their formal mathematical treatment for this sheet.

### Worked Example 1: Nested Loop

```python
n = 5
for i in range(n):
    for j in range(n):
        print(i, j)
```

The outer loop runs `n` times. For **every single one** of those `n` outer iterations, the inner loop also runs a full `n` times. Total operations: `n × n = n²`.

**Time complexity: O(n²)**

---

### Worked Example 2: Triangular Nested Loop

```python
n = 5
for i in range(n):
    for j in range(i + 1):
        print(i, j)
```

This time the inner loop's range depends on the outer loop's current value — it runs `i + 1` times for each `i`. Across all outer iterations, the total number of inner-loop executions is:

`1 + 2 + 3 + ... + n`

This is the classic sum of the first `n` natural numbers, which has a known closed form: `n(n+1)/2`, which expands to `n²/2 + n/2`.

Applying the three rules: drop the lower-order term `n/2` (Rule 3), leaving `n²/2` — then drop the constant coefficient `1/2` too.

**Time complexity: O(n²)**

Notice both examples land on O(n²) despite doing visibly different amounts of actual work (25 total prints in example 1 vs. 15 in example 2, for `n=5`) — Big O cares about the _growth rate_, not the exact operation count, and both of these scale quadratically as `n` increases.

## What Is Space Complexity?

Space complexity measures how much **memory** your code requires, again expressed independent of the machine — using the same Big O notation, rather than absolute units like MB or GB.

**Space complexity = input space + auxiliary space.**

- **Input space** is the memory used to store the inputs themselves.
- **Auxiliary space** is any _extra_ memory your code uses beyond that, to actually solve the problem.

```python
def add(a, b):
    c = a + b
    return c
```

Here, `a` and `b` occupy input space, and `c` is auxiliary space — one extra variable. Since this doesn't grow no matter how large `a` and `b` get (it's always exactly one extra variable), this is **O(1)** — constant space.

Compare this to a function that builds a new array proportional to its input:

```python
def create_array(n):
    arr = [0] * n
    return arr
```

Here, the auxiliary space grows directly with `n` — twice the input means twice the memory for `arr`. This is **O(n)** space complexity.

### Good Practice: Don't Mutate the Input to "Save" Space

Recall from section 1.7 that mutable types like lists can be changed in place. It might occur to you that you can reduce a function's memory usage by modifying the input directly instead of creating a new structure:

```python
# Technically saves space, but modifies the caller's original data
def double_in_place(arr):
    for i in range(len(arr)):
        arr[i] *= 2
    return arr
```

**Avoid this unless you're explicitly told to.** Mutating input data that doesn't belong to your function is considered poor practice, both in interviews and in real production code — the caller may need that original data afterward for something else entirely, and silently changing it out from under them can introduce hard-to-trace bugs elsewhere in a larger system. The safer default is to build and return a new structure, even at the cost of extra space:

```python
# Preserves the original — the correct default behavior
def double_copy(arr):
    return [x * 2 for x in arr]
```

**Only mutate the input directly if a problem statement or interviewer explicitly permits or requests it.** Until then, treat "don't touch what was handed to you" as the default rule.

### Time Limits on Judges — and Why This Number Is Different for Python

A commonly cited rule of thumb on competitive programming platforms is that a typical judge server executes roughly **10⁸ operations per second**. So if a problem gives you a 2-second time limit, your solution's total operation count should stay around `2 × 10⁸`, not far beyond it.

**This rule of thumb was calibrated for compiled languages like C++, and it does not transfer directly to Python.** Python is an interpreted language, and even simple operations carry noticeably more per-step overhead than their compiled-language equivalents. A realistic, conservative estimate for pure Python loops is closer to **10⁶–10⁷ operations per second** — roughly one to two orders of magnitude slower than the C++ benchmark.

Practical implications worth internalizing:

- **Many online judges (Codeforces, for example) give Python solutions a higher time-limit multiplier** specifically to account for this — always check the platform's stated policy rather than assuming the raw time limit applies equally across languages.
- **When estimating whether your algorithm will run in time**, be more conservative with Python than the O(10⁸) rule suggests — treat O(10⁷) as a safer ceiling for a 1-second budget in pure Python, and be cautious with anything approaching O(n²) once `n` exceeds roughly 10,000.
- **Vectorized operations (via libraries like NumPy) run drastically faster than equivalent pure Python loops**, since the heavy lifting happens in compiled C code under the hood — but this is a separate topic from algorithmic complexity, and won't come up until later, more applied parts of your ML/data work.

### Bonus: Time Complexity of Common Python Built-in Operations

This is genuinely essential, practical knowledge that isn't always spelled out clearly, and directly affects how you write efficient solutions from here on. The complexity of your _algorithm_ on paper means nothing if you unknowingly use a built-in operation with a hidden cost.

**Lists:**

|Operation|Code|Time Complexity|
|---|---|---|
|Access by index|`arr[i]`|O(1)|
|Set by index|`arr[i] = x`|O(1)|
|Append to end|`arr.append(x)`|O(1) amortized|
|Pop from end|`arr.pop()`|O(1)|
|Insert at index|`arr.insert(i, x)`|O(n)|
|Pop at arbitrary index|`arr.pop(i)`|O(n)|
|Remove by value|`arr.remove(x)`|O(n)|
|Membership check|`x in arr`|O(n)|
|Slice|`arr[a:b]`|O(b - a)|
|Sort|`arr.sort()`|O(n log n)|
|Length|`len(arr)`|O(1)|

**Dictionaries and Sets** (both hash-based):

|Operation|Code|Time Complexity|
|---|---|---|
|Get value|`d[key]`|O(1) average|
|Set value|`d[key] = x`|O(1) average|
|Delete key|`del d[key]`|O(1) average|
|Membership check|`key in d` or `x in some_set`|O(1) average|

The contrast between list membership (`in`) at **O(n)** versus set/dict membership at **O(1) average** is one of the most consequential things you can internalize early — it's the direct, practical reason hashing (covered later in this chapter) exists and matters so much. If you ever find yourself repeatedly checking `x in some_list` inside a loop, that's a strong signal you should be using a set instead.

### Bonus: Measuring Actual Runtime (When You Want To)

While complexity analysis is what matters for algorithm design, sometimes you genuinely want to measure real elapsed time — for comparing two working implementations empirically, for instance. Python's `timeit` module handles this properly (it accounts for things simple `time.time()` calls can get wrong, like warming up the interpreter):

```python
import timeit

n = 1000
elapsed = timeit.timeit(lambda: sum(range(n)), number=1000)
print(f"Total time for 1000 runs: {elapsed:.6f} seconds")
```

Treat this as a tool for empirical comparison, not a substitute for understanding Big O — actual runtime is still machine-dependent, exactly as discussed at the start of this section.

|Concept|Key Idea|
|---|---|

## Common Pitfalls to Avoid

- **Timing your code and calling that "the time complexity."** Actual runtime depends on hardware; complexity is about growth rate relative to input size.
- **Forgetting Python's built-in operations have their own hidden costs.** `x in some_list` is O(n) even though it looks like a simple one-line check — swap to a set if you're doing this repeatedly.
- **Applying the C++-calibrated "10⁸ operations per second" rule directly to Python** without adjusting your expectations downward.
- **Mutating input data to save space "because it's more efficient."** Efficiency doesn't excuse silently modifying data that isn't yours to change — extra space is usually the correct tradeoff.
- **Stopping analysis at best case or average case for interview purposes.** Worst case is what's expected unless explicitly stated otherwise.
