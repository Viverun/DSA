### Quick Primer: Defining a Function in Python

Before getting into how arguments behave when passed around, a fast refresher on the syntax itself, since this is your first real look at functions in this sheet:

```python
def add(a, b):
    result = a + b
    return result

print(add(3, 5))
```

**Output:**

```
8
```

- `def` starts a function definition, followed by a name and parentheses containing **parameters** (the inputs the function expects).
- The colon and indented block work exactly like `if` and loops — indentation defines what belongs to the function.
- `return` sends a value back to wherever the function was called from. A function with no `return` statement implicitly returns `None`.

With that refresher out of the way — the real subject of this topic, and one of the most important mental models you'll build in this entire sheet: **what actually happens to the data you pass into a function?**

---

### The Core Question

When you hand a variable to a function, does the function get its own independent copy, or does it get access to the _actual_ original data? The answer changes everything about whether modifying something inside a function affects the caller's variable — and getting this wrong is one of the most common sources of confusing bugs for anyone new to Python.

---

### Python's Actual Model: Pass by Object Reference

Here's the concept to really absorb: in Python, every variable is a **name that points to an object** sitting somewhere in memory. It's less "a box holding a value" and more "a label attached to a value." When you pass a variable into a function, Python passes **a reference to that same object** — not a brand-new copy, and not the variable itself either. This behavior has a specific name: **pass by object reference** (sometimes called "pass by assignment").

What this means in practice depends entirely on **one crucial fact about the object being passed: is it mutable or immutable?**

---

### Immutable Types Behave Like Pass by Value

Immutable types — `int`, `float`, `str`, `bool`, `tuple` — can never be changed in place once created (you saw this already for strings in 1.2). Any operation that looks like it modifies one of these actually creates a **brand new object**.

```python
def modify(a):
    a = a + 10

x = 5
modify(x)
print(x)   # 5 — unchanged
```

**Output:**

```
5
```

Here's what's really happening: inside `modify`, the line `a = a + 10` doesn't change the number `5` — it **creates a new integer object**, `15`, and rebinds the local name `a` to point at it. The caller's `x` was never touched; it still points at the original `5` the entire time. Once `modify()` finishes, that local `a` disappears completely, and `x` outside is exactly as it was.

---

### Mutable Types Behave Like Pass by Reference

Mutable types — `list`, `dict`, `set`, and custom objects by default — **can** be changed in place, without creating a new object. When you pass one of these into a function and the function mutates it _in place_ (rather than reassigning it), the change is visible to the caller, because both the caller's variable and the function's parameter point to the exact same object in memory.

```python
def modify(lst):
    lst.append(10)

nums = [5]
modify(nums)
print(nums)   # [5, 10] — changed!
```

**Output:**

```
[5, 10]
```

`lst.append(10)` doesn't create a new list — it modifies the existing list object directly. Since `nums` and `lst` are two different names pointing at that same single list, the change shows up no matter which name you look at it through afterward.

---

### The Real Mechanism: Rebinding vs Mutating

This is, without exaggeration, the single most important distinction in this entire topic — internalize it fully, because it explains every confusing case you'll run into later:

- **Rebinding** a name (`a = something_new`) makes that name point to a _different_ object. It never affects anything else that was pointing at the old object — including the caller's variable.
- **Mutating** an object (`lst.append(x)`, `lst[0] = 5`, `lst.sort()`) changes the object _itself_, in place. Every name currently pointing at that object sees the change, because there's only ever been one object the whole time.

The clearest way to see this is to compare two functions that look almost identical but behave completely differently:

```python
def reassign(lst):
    lst = [99, 99, 99]   # REBINDING — lst now points to a brand new list

def mutate(lst):
    lst.append(99)        # MUTATING — the original list itself is changed

original = [1, 2, 3]
reassign(original)
print(original)   # [1, 2, 3] — unchanged! reassign only redirected its local name

original = [1, 2, 3]
mutate(original)
print(original)   # [1, 2, 3, 99] — changed! mutate touched the shared object
```

**Output:**

```
[1, 2, 3]
[1, 2, 3, 99]
```

Both functions received the same kind of object (a mutable list) — the difference in outcome is entirely about _what the function did with it_: pointed its local name somewhere new, versus reached into the shared object and changed its contents.

---

### Proving It with `id()`

Python's built-in `id()` function returns a unique identifier for an object — effectively its memory address. This is a genuinely useful way to _see_ the rebinding-vs-mutating distinction directly instead of just trusting the explanation.

```python
def show_and_modify(a):
    print("Inside before:", id(a))
    a = a + 10
    print("Inside after: ", id(a))

x = 5
print("Outside:       ", id(x))
show_and_modify(x)
```

**Output (actual numbers will vary by run):**

```
Outside:        4380372784
Inside before:  4380372784
Inside after:   4380373104
```

Notice: "Outside" and "Inside before" show the **same** id — confirming `a` initially points to the exact same object as `x`. But after `a = a + 10`, the id changes — proving a completely new object was created, and `x` outside was never involved.

Now the mutable version:

```python
def show_and_append(lst):
    print("Inside before:", id(lst))
    lst.append(10)
    print("Inside after: ", id(lst))

nums = [5]
print("Outside:       ", id(nums))
show_and_append(nums)
```

**Output (actual numbers will vary by run):**

```
Outside:        140312657573952
Inside before:  140312657573952
Inside after:   140312657573952
```

Here, the id **never changes** — before the append, after the append, and outside the function, it's all the same single list object the whole time. That's the entire mechanism, made visible.

### Forcing Value-Like Behavior for Mutable Types

Sometimes you deliberately _want_ to protect the caller's original list, dict, or set from being modified by a function — you want value-like safety even though the type is mutable. The fix is to pass (or work on) an explicit **copy** instead of the original:

```python
def modify_safely(lst):
    lst = lst.copy()   # now working on an independent copy
    lst.append(10)
    return lst

original = [5]
new_list = modify_safely(original)
print(original)   # [5] — untouched
print(new_list)    # [5, 10] — the modified copy
```

Be aware this only makes a **shallow copy** — if your list contains other mutable objects nested inside it (a list of lists, for instance), the inner objects are still shared between the copy and the original. For a fully independent copy of a nested structure, use `copy.deepcopy()`:

```python
import copy

nested = [[1, 2], [3, 4]]
deep_copy = copy.deepcopy(nested)
deep_copy[0].append(99)

print(nested)     # [[1, 2], [3, 4]] — untouched
print(deep_copy)   # [[1, 2, 99], [3, 4]]
```

### Returning Multiple Values — Python's Substitute for "Reference Parameters"

In languages with true reference parameters, one common use is updating several variables at once through a function, or effectively "returning" more than one value. Python doesn't offer reference parameters for this — but it doesn't need to, because Python functions can just **return a tuple**, and the idiomatic way to "return multiple values" is to unpack it directly at the call site:

```python
def compute(a, b):
    return a + b, a - b, a * b

total, difference, product = compute(5, 3)
print(total)        # 8
print(difference)   # 2
print(product)      # 15
```

This is genuinely the standard, preferred way to get multiple results out of a Python function — reach for this pattern rather than trying to mimic reference-parameter tricks from other languages.

## Quick Reference: Immutable vs Mutable Types

|Immutable (behave like pass-by-value)|Mutable (behave like pass-by-reference)|
|---|---|
|`int`|`list`|
|`float`|`dict`|
|`str`|`set`|
|`bool`|custom objects/classes (by default)|
|`tuple`||
|`frozenset`||

A useful trick to remember which category a type falls into: if it has methods that change it _in place_ without needing a `return` (like `.append()`, `.sort()`, `.add()`), it's mutable. If every operation on it that "changes" it actually needs to be reassigned back to a variable (`s = s.upper()`, `x = x + 1`), it's immutable.

## Common Pitfalls to Avoid

- **Assuming all objects behave the same way when passed to a function.** It depends entirely on whether the type is mutable or immutable — there's no single universal rule to memorize other than "check the type."
- **Confusing rebinding with mutating.** `lst = [...]` inside a function never affects the caller. `lst.append(...)`, `lst[i] = ...`, `lst.sort()` all affect the caller (for mutable types). This distinction explains nearly every "why didn't my function's changes show up outside it?" bug you'll hit.
- **Accidentally mutating a caller's list when you meant to work on a copy.** If a function shouldn't change the original, explicitly copy it first (`lst = lst.copy()`) before mutating.
- **Using a shallow copy when a deep copy was actually needed**, for lists containing nested mutable objects — the outer list gets copied, but inner lists/dicts are still shared.
- **Trying to simulate reference-parameter tricks from other languages.** Python's idiomatic way to get multiple outputs from a function is returning a tuple, not attempting to pass something "by reference."

My Question: If every time we are renaming a variable(Immutable type ) and it creates a new object and point toward that but the older one remain unchanged then eventually isn't this quite memory consuming as the original object and new object both exist in the system

Excellent question. This is exactly the kind of thing Python's memory management is designed to handle.

The short answer is: **temporarily yes, but not permanently.**

Let's go through it.

### Example

```python
a = 10
```

Memory:

```
a ─────► 10
```

Now rebind:

```python
a = 20
```

Memory immediately after rebinding:

```
20 ◄───── a

10      (no variable points here anymore)
```

The old integer `10` hasn't necessarily disappeared the instant you assign `20`. It simply has **no references** (assuming nothing else was pointing to it).

Python keeps track of how many references each object has.

```
Object   References
10           0
20           1
```

Once an object's reference count becomes **0**, Python knows nothing in your program can ever reach it again.

It becomes eligible for destruction.

---

## Does memory keep increasing?

No.

Python automatically frees objects that are no longer referenced.

```python
a = 10
a = 20
a = 30
a = 40
```

You are **not** storing four integers forever.

After each reassignment:

```
10 → reference count = 0 → deleted
20 → reference count = 0 → deleted
30 → reference count = 0 → deleted
40 → reference count = 1
```

Only the current value survives (ignoring some implementation optimizations discussed below).

---

## Then why doesn't Python mutate integers?

Imagine if integers were mutable.

```python
a = 5
b = a

a += 1
```

If integers were mutable:

```
Before:

a ─┐
   ▼
   5
   ▲
b ─┘
```

Changing `a` would change the shared object:

```
6
▲ ▲
│ │
a b
```

Now `b` unexpectedly becomes `6`.

That's a disaster.

Instead, Python creates a new integer object:

```
a ─► 6

b ─► 5
```

This makes immutable objects predictable and safe.

---

## But for large objects isn't this expensive?

Suppose

```python
s = "A" * 1_000_000
```

Now

```python
s = s + "B"
```

A **new string** is created because strings are immutable.

For a brief moment:

```
Old 1 MB string
New 1 MB + 1 byte string
```

Both exist simultaneously.

Then the old one loses all references.

Python frees it.

So yes, peak memory briefly increases, but not forever.

---

## Python also optimizes this

Python doesn't blindly create duplicate objects.

### Small integer caching

```python
a = 10
b = 10
```

Usually:

```
a ─┐
   ▼
  10
   ▲
b ─┘
```

Only one integer object exists.

---

### String interning

```python
a = "hello"
b = "hello"
```

Often both point to the same string object.

Again, memory is saved.

---

## Mutable objects are different

```python
lst = [1, 2]

a = lst
b = lst

a.append(3)
```

Memory:

```
a ─┐
   ▼
[1,2,3]
   ▲
b ─┘
```

Only one list exists.

Appending does **not** create another list.

---

## Why isn't immutability wasteful?

Because objects that become unreachable are reclaimed automatically. The trade-off is generally worth it because immutable objects provide:

- Simpler reasoning (no unexpected changes through aliases)
    
- Safer sharing between variables, functions, and threads
    
- Optimizations like object caching and interning
    
- Reliable hashing (which enables their use as dictionary keys and set elements)
    

The temporary extra memory used during rebinding is usually small and short-lived.

### Mental model

Think of variables as sticky notes on boxes:

```
Initially

Sticky note "a"
      │
      ▼
   [ Box: 10 ]
```

After:

```python
a = 20
```

You don't rewrite the box. You move the sticky note:

```
Sticky note "a"
      │
      ▼
   [ Box: 20 ]

[ Box: 10 ]   ← no sticky notes
```

The cleaning crew (Python's garbage collector/reference counting) notices that no sticky notes point to the old box anymore and throws it away.

So while rebinding an immutable object can briefly result in both the old and new objects existing at the same time, Python automatically reclaims the old one once nothing references it, preventing memory from growing indefinitely.
