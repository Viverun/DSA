### Does Python Have a Switch Statement?

For a long time, the honest answer was **no** — Python simply didn't have a switch/case construct at all. That changed with **Python 3.10**, which introduced the `match-case` statement (formally called "structural pattern matching"). It fills the same role as switch-case, and it's actually more powerful.

But because `match-case` is relatively recent, you'll also frequently see — and should know how to write — the classic, universally-compatible Python alternative: **dictionary mapping**. Both are covered here, since you'll run into both in real code and on different judges/environments.

---

## Approach 1: Dictionary Mapping (Works on Any Python Version)

The core idea: instead of writing a chain of comparisons against a single variable, you store the mapping of "value → result" directly in a dictionary, then just look up the key.

```python
day = int(input("Enter a number (1-7): "))

days = {
    1: "Monday",
    2: "Tuesday",
    3: "Wednesday",
    4: "Thursday",
    5: "Friday",
    6: "Saturday",
    7: "Sunday"
}

print(days.get(day, "Invalid"))
```

**Input:** `4` **Output:**

```
Thursday
```

### Handling the "Default" Case: `.get()`

The `.get(key, default)` method is doing the heavy lifting here — it looks up `key` in the dictionary, and if the key doesn't exist, it returns `default` instead of raising an error. This is your equivalent of a switch statement's `default:` case, in one clean expression.

Compare that to what happens if you use plain `days[day]` instead:

```python
print(days[10])   # KeyError: 10
```

Direct indexing with `[]` raises a `KeyError` if the key is missing — `.get()` with a fallback is almost always what you want when a "no match" case is expected and valid.

This dictionary-mapping pattern is genuinely idiomatic Python, not just a workaround — you'll see it used in real production code specifically _because_ it's clean, and it also naturally generalizes to mapping values to **function calls** (a more advanced dispatch pattern you'll encounter later once you're comfortable with functions).

## Approach 2: `match-case` (Python 3.10+)

If you're running Python 3.10 or newer, `match-case` is the direct, purpose-built equivalent — and it reads very close to traditional switch syntax.

```python
day = int(input("Enter a number (1-7): "))

match day:
    case 1:
        print("Monday")
    case 2:
        print("Tuesday")
    case 3:
        print("Wednesday")
    case 4:
        print("Thursday")
    case 5:
        print("Friday")
    case 6:
        print("Saturday")
    case 7:
        print("Sunday")
    case _:
        print("Invalid")
```

**Input:** `4` **Output:**

```
Thursday
```

### The Default Case: the Underscore Wildcard `_`

`case _:` is the `match-case` equivalent of `default:` — the underscore is a wildcard pattern that matches literally anything, so it's always placed **last**, catching whatever none of the earlier cases matched.

### A Genuine Improvement: No `break`, No Fall-Through

This is worth calling out explicitly because it removes an entire category of classic switch-statement bugs. In C-style switch statements, forgetting a `break` causes execution to silently "fall through" into the next case. **`match-case` has no such behavior** — once a case matches and its block runs, Python exits the `match` block automatically. You never need `break`, and you can never accidentally fall through by forgetting one.

### Bonus: Matching Multiple Values to One Case

Another improvement — grouping several values into a single case is a one-liner with `|` (the "or" pattern), instead of writing separate stacked cases:

```python
match day:
    case 6 | 7:
        print("Weekend")
    case _:
        print("Weekday")
```

## Matching on Computed Expressions

Both approaches handle a computed expression (not just a bare variable) exactly as naturally as a plain value — there's no special restriction here in Python the way there is in some languages.

```python
x = 10
y = 5

match x + y:
    case 15:
        print("Result is 15.")
    case 20:
        print("Result is 20.")
    case _:
        print("No match found.")
```

**Output:**

```
Result is 15.
```

The dictionary version of the same logic:

```python
x = 10
y = 5

results = {15: "Result is 15.", 20: "Result is 20."}
print(results.get(x + y, "No match found."))
```

## Matching Strings and Characters

Python places **no restriction** on what type you can match against — unlike switch statements limited to integers or characters, `match-case` (and dictionary keys) work identically with strings, and in fact with almost any data type, including tuples and more complex structures you'll encounter later.

```python
grade = "B"

match grade:
    case "A":
        print("Excellent!")
    case "B":
        print("Good!")
    case _:
        print("Not specified.")
```

**Output:**

```
Good!
```

Dictionary version:

```python
grade = "B"

responses = {"A": "Excellent!", "B": "Good!"}
print(responses.get(grade, "Not specified."))
```

## Bonus: Guard Clauses in `match-case`

One more capability worth knowing, since it goes beyond what a traditional switch can do at all: you can attach an extra condition to a case using `if`, called a **guard clause**. The case only matches if both the pattern _and_ the guard condition are true.

```python
marks = 82

match marks:
    case m if m >= 90:
        print("Grade: A+")
    case m if m >= 75:
        print("Grade: A")
    case m if m >= 50:
        print("Grade: B")
    case _:
        print("Grade: F")
```

**Output:**

```
Grade: A
```

This starts to blur the line between `match-case` and a regular `if-elif` chain — and that's fine. Use whichever reads more clearly for the problem at hand. `match-case` tends to shine specifically when you're comparing one variable against a set of exact, discrete values (closer to the classic switch use case); reach for `if-elif` when your conditions are ranges or more complex boolean logic (as in section 1.3).

## Cheat Sheet — 1.4 Switch Case Equivalent

|Task|Dictionary Mapping|`match-case` (3.10+)|
|---|---|---|
|Basic value matching|`d = {1: "A", 2: "B"}` then `d.get(x, "default")`|`match x: case 1: ... case _: ...`|
|Default/fallback case|`.get(key, default)`|`case _:`|
|Multiple values → same result|repeat key-value pairs, or build the dict accordingly|`case 6|
|Matching on an expression|`d.get(x + y, "default")`|`match x + y:`|
|Matching strings|dict keys can be strings directly|`case "B":`|
|Extra condition on a match|not directly supported — use `if` inside|`case m if m >= 90:`|

## When to Use Which

- **Dictionary mapping** — safest default choice, works on every Python version, and is especially natural when the "result" for each case is itself a value (a string, a number) rather than a multi-line block of logic.
- **`match-case`** — reach for this when you're on Python 3.10+, the branching logic per case is more than a single value (multiple statements, side effects), or you want the guard-clause / multi-value / structural-matching power it offers.
- **Plain `if-elif`** (from 1.3) — still the right tool when conditions involve ranges, complex boolean combinations, or comparisons that aren't simple "does this equal one of these exact values" checks.

---

## Common Pitfalls to Avoid

- **Assuming `match-case` is available everywhere.** It requires Python 3.10+. If you're unsure what version an online judge or grading environment is running, the dictionary approach is the safer bet.
- **Using `d[key]` instead of `d.get(key, default)`** when a missing key is a valid, expected scenario — `d[key]` will crash with a `KeyError` instead of gracefully falling back.
- **Forgetting `case _:` as the last case.** Without it, if nothing matches, the `match` block just does nothing silently — no error, no output — which can be confusing to debug if you expected a default message.
- **Over-using guard clauses until `match-case` is just a disguised `if-elif` chain.** If every case has a guard, you've reinvented `if-elif` with extra syntax — just use `if-elif` at that point.
