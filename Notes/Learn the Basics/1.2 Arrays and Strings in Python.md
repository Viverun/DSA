## Part A: Arrays

### 1. What Is an Array, Conceptually

An array is a **linear data structure** used to store multiple values together, where each value sits in a specific position (an _index_), and that position lets you jump straight to the value without searching for it — this is called **random access**.

Conceptually, think of memory as a long row of boxes. An array claims a stretch of consecutive boxes and puts one value in each. Because the boxes are next to each other, the computer can calculate exactly where the 5th box is without walking through boxes 1–4 first. That's what makes indexed access fast — **O(1), constant time** — regardless of how large the array is.

This contiguous, indexed layout is _the_ foundational idea behind almost every data structure you'll study after this. Get comfortable with it now.

---

### 2. Python's Array Equivalent: The List

Python's primary tool for this job is the **list**. You'll use it constantly — it is, without exaggeration, the single most-used data structure in this entire sheet.

```python
arr = [10, 20, 30, 40, 50]
print(arr)
```

**Output:**

```
[10, 20, 30, 40, 50]
```

A quick honest note: Python lists are more flexible than the traditional "array" concept — they can grow and shrink freely, and technically they can hold mixed types in the same list. In DSA work, though, you'll almost always keep a list **homogeneous** (all integers, or all strings, etc.), both because problems expect it and because mixing types breaks most algorithms (you can't meaningfully sort `[3, "apple", True]`, for instance).

There's also a stricter, typed `array` module in Python's standard library (`from array import array`) that behaves closer to a traditional fixed-type array. You won't need it for this sheet — lists are the standard tool for every array-based DSA problem you'll solve.

---

### 3. Zero-Based Indexing

Like most programming languages, Python indexes from **0**, not 1.

```python
arr = [10, 20, 30, 40, 50]
print(arr[0])   # first element
print(arr[4])   # last element
```

**Output:**

```
10
50
```

A list of length `n` has valid indices from `0` to `n-1`. Trying to access `arr[5]` here would raise an `IndexError: list index out of range`. This is one of the most common runtime errors you will hit while solving problems — always double check your loop bounds.

**Python-specific bonus:** negative indexing. `-1` means "last element," `-2` means "second-to-last," and so on. This is genuinely useful and you'll use it constantly:

```python
print(arr[-1])   # 50 (last element)
print(arr[-2])   # 40 (second-to-last)
```

---

### 4. Creating a List

A few common ways you'll create lists throughout this sheet:

```python
# Empty list, to be filled later
arr = []

# List with known values
arr = [1, 2, 3, 4, 5]

# List of n zeros (very common — pre-sizing before filling in a loop)
n = 5
arr = [0] * n
print(arr)   # [0, 0, 0, 0, 0]

# List built from user input (covered in 1.1)
arr = list(map(int, input().split()))

# List generated with a range
arr = list(range(1, 6))
print(arr)   # [1, 2, 3, 4, 5]
```

---

### 5. Memory Behavior: Dynamic, Not Fixed-Size

You do **not** declare a size upfront in Python the way some languages require. A list grows and shrinks as needed — internally, Python over-allocates extra space behind the scenes so that adding elements is fast on average, but as far as you're concerned, you just add and remove freely:

```python
arr = [1, 2, 3]
arr.append(4)
print(arr)   # [1, 2, 3, 4]
```

This convenience is worth appreciating, but don't let it make you sloppy about complexity analysis — operations still have real costs depending on _where_ in the list you're adding or removing (covered in section 9).

---

### 6. Accessing Elements — Indexing & Negative Indexing

Already shown above, but worth stating as a clean rule:

|Syntax|Meaning|
|---|---|
|`arr[0]`|first element|
|`arr[i]`|element at index `i`|
|`arr[-1]`|last element|
|`arr[len(arr)-1]`|last element (equivalent, more verbose)|

Both direct access and modification are **O(1)**:

```python
arr = [10, 20, 30]
arr[1] = 99
print(arr)   # [10, 99, 30]
```

---

### 7. Slicing

Slicing is one of Python's most powerful array features, and it doesn't have a clean equivalent in a lot of other languages — you'll rely on it heavily once you get into array problems (subarrays, reversing, windows, etc.).

Syntax: `arr[start:stop:step]` — `start` is inclusive, `stop` is exclusive.

```python
arr = [10, 20, 30, 40, 50]

print(arr[1:4])     # [20, 30, 40]  → indices 1,2,3
print(arr[:3])      # [10, 20, 30]  → from start up to index 3
print(arr[2:])      # [30, 40, 50]  → from index 2 to end
print(arr[:])       # [10, 20, 30, 40, 50]  → full copy
print(arr[::-1])    # [50, 40, 30, 20, 10]  → reversed
print(arr[::2])     # [10, 30, 50]  → every 2nd element
```

`arr[::-1]` in particular is a pattern you'll use constantly for reversing arrays and strings — memorize it now.

**Important:** slicing creates a **new list** (a copy), it doesn't just give you a view into the old one. This matters for both correctness and performance — slicing a huge list repeatedly inside a loop can quietly cost you more time than expected.

---

### 8. Searching for an Element

Three approaches, matching the three general strategies for finding data:

**a) Direct index (when you already know the position):**

```python
arr = [10, 20, 30, 40, 50]
print(arr[2])   # 30
```

**b) Membership check with `in` (does the value exist at all?):**

```python
print(30 in arr)   # True
print(99 in arr)   # False
```

This is a linear scan under the hood — **O(n)**.

**c) Find the index of a value with `.index()`:**

```python
print(arr.index(30))   # 2
```

Also **O(n)**, and raises a `ValueError` if the value isn't found — so guard with `in` first if you're not sure it exists, or wrap in a `try/except`.

For repeated fast lookups (checking membership over and over), you'll later switch to **sets** or **dictionaries**, which give O(1) average lookup instead of O(n) — that's covered in the Hashing section of this chapter. Keep it in mind as a preview of _why_ hashing matters.

---

### 9. Inserting & Removing Elements

|Operation|Code|Time Complexity|Notes|
|---|---|---|---|
|Add to end|`arr.append(x)`|O(1) amortized|Most common, most efficient|
|Insert at specific index|`arr.insert(i, x)`|O(n)|Everything after index `i` shifts right|
|Remove from end|`arr.pop()`|O(1)|Removes and returns last element|
|Remove at specific index|`arr.pop(i)`|O(n)|Everything after index `i` shifts left|
|Remove by value|`arr.remove(x)`|O(n)|Removes first matching value; error if not found|
|Delete by index|`del arr[i]`|O(n)|Same shifting cost as `pop(i)`, but no return value|

```python
arr = [1, 2, 3]
arr.append(4)        # [1, 2, 3, 4]
arr.insert(1, 99)     # [1, 99, 2, 3, 4]
arr.pop()             # removes 4 → [1, 99, 2, 3]
arr.pop(1)             # removes 99 → [1, 2, 3]
arr.remove(2)          # removes value 2 → [1, 3]
```

The takeaway to internalize: **operations at the end of a list are cheap (O(1)); operations in the middle or start are expensive (O(n))** because Python has to physically shift every following element. This is exactly the same underlying cost model as the traditional array — Python's convenience methods don't change the fundamental complexity.

---

### 10. A Critical Pitfall: Assignment Doesn't Copy

This catches almost everyone coming from languages with value-type arrays. In Python, lists are **reference types**. Assigning one list to another variable does _not_ create a new list — both names point to the _same_ underlying list in memory.

```python
a = [1, 2, 3]
b = a          # b is NOT a copy — it's the same list as a
b.append(4)
print(a)       # [1, 2, 3, 4]  ← a changed too!
```

To actually copy a list, be explicit:

```python
b = a.copy()          # shallow copy
b = a[:]               # shallow copy via full slice
b = list(a)             # shallow copy via constructor
```

This will bite you eventually if you're not aware of it now — especially when passing lists into functions or storing them inside other structures. Keep this in the back of your mind for the whole sheet.

## Part B: Strings

### 1. What Is a String

A string is a **sequence of characters**, and — importantly — in Python, a string _is itself_ a kind of ordered, indexed sequence, very similar in behavior to a list of characters, with one enormous difference covered in section 5.

```python
s = "striver"
```

---

### 2. Zero-Based Indexing (Strings)

Exactly the same indexing rules as lists apply:

```python
s = "striver"
print(s[0])   # 's'
print(s[1])   # 't'
print(s[2])   # 'r'
print(s[-1])  # 'r' (last character)
```

---

### 3. Length

```python
s = "striver"
print(len(s))   # 7
```

`len()` is the universal length function in Python — it works on strings, lists, dictionaries, sets, and more. One function, consistent behavior everywhere.

---

### 4. Accessing Characters & Slicing

Everything you learned about list slicing applies identically to strings:

```python
s = "striver"

print(s[1:4])     # 'tri'
print(s[:3])      # 'str'
print(s[3:])      # 'iver'
print(s[::-1])    # 'reviirts'  → reversed string, extremely common pattern
```

`s[::-1]` is the standard, idiomatic way to reverse a string in Python. You will use this constantly — palindrome checks are the most obvious early example.

---

### 5. Immutability — The Core Difference From Lists

This is the single most important thing to understand about Python strings, and it's a genuine behavioral difference from lists, not just a syntax detail.

**Strings cannot be modified in place.** Once created, a string's characters are fixed. Any operation that looks like it's "changing" a string is actually building a **brand new string** in memory.

```python
s = "striver"
s[0] = "S"   # This will raise: TypeError: 'str' object does not support item assignment
```

If you need to change a character, you build a new string instead:

```python
s = "striver"
s = "S" + s[1:]
print(s)   # 'Striver'
```

**Why this matters for DSA specifically:** if a problem requires lots of character-by-character modification (building a string piece by piece in a loop, for example), repeatedly doing `s = s + char` is quietly expensive — each `+` creates an entirely new string copy, which makes a naive loop **O(n²)** instead of the O(n) you'd expect. The standard workaround:

```python
# Inefficient pattern — avoid in loops over large input:
s = ""
for ch in "striver":
    s += ch

# Efficient pattern — build a list, join once at the end:
chars = []
for ch in "striver":
    chars.append(ch)
s = "".join(chars)
```

You won't feel this cost on small strings, but it's a habit worth building early — it becomes a real performance issue once inputs scale up later in the sheet.

---

### 6. Concatenation

```python
a = "Hey, "
b = "Striver!"
print(a + b)          # 'Hey, Striver!'

# f-strings (preferred when mixing text and variables — see 1.1)
name = "Striver"
print(f"Hey, {name}!")   # 'Hey, Striver!'

# join() — the efficient way to combine many pieces at once
words = ["Hey", "Striver"]
print(", ".join(words))   # 'Hey, Striver'
```

---

### 7. Passing Strings Around

Because strings are immutable, there's no risk of a function accidentally modifying a string you pass into it — unlike lists, where a function _can_ mutate the original list you hand it (since lists are reference types, as covered above). Any "modification" inside a function on a string is really just that function building and returning a new string. This makes strings inherently safer to pass around without unexpected side effects.

```python
def shout(text):
    text = text.upper()   # creates a new string, doesn't touch the caller's original
    return text

original = "striver"
result = shout(original)
print(original)   # 'striver'  ← unchanged
print(result)      # 'STRIVER'
```

---

### 8. Comparison Operators

Strings compare the same way numbers do, using `==` and `!=`, and Python compares them **lexicographically** (dictionary order, based on character codes):

```python
print("apple" == "apple")   # True
print("apple" != "banana")  # True
print("apple" < "banana")   # True  → 'a' comes before 'b'
print("Apple" == "apple")   # False → case-sensitive
```

Comparing individual characters works exactly the same:

```python
s1 = "striver"
s2 = "strive"
print(s1[0] == s2[0])   # True → both 's'
```

---

### 9. Useful String Methods for DSA

A working toolkit you'll reach for repeatedly across this sheet:

|Method|What it does|Example|
|---|---|---|
|`.split()`|splits into a list on whitespace (or a given separator)|`"a b c".split()` → `['a','b','c']`|
|`.join(list)`|joins a list of strings into one, using this string as separator|`",".join(['a','b'])` → `'a,b'`|
|`.strip()`|removes leading/trailing whitespace|`" hi ".strip()` → `'hi'`|
|`.lower()` / `.upper()`|case conversion|`"Hi".lower()` → `'hi'`|
|`.replace(old, new)`|replaces all occurrences|`"aab".replace("a","x")` → `'xxb'`|
|`.find(sub)`|index of first match, or `-1` if not found|`"striver".find("ri")` → `2`|
|`.count(sub)`|counts occurrences|`"banana".count("a")` → `3`|
|`.isalpha()`|True if all characters are letters|`"abc".isalpha()` → `True`|
|`.isdigit()`|True if all characters are digits|`"123".isdigit()` → `True`|
|`list(s)`|converts string into a list of characters|`list("abc")` → `['a','b','c']`|

That last one — `list(s)` — is your escape hatch whenever a problem _does_ require character-by-character mutation: convert to a list, mutate freely (lists are mutable), then `"".join(...)` back into a string when you're done.

```python
s = "striver"
chars = list(s)
chars[0] = "S"
s = "".join(chars)
print(s)   # 'Striver'
```

## Common Pitfalls to Avoid

- **`IndexError`**: accessing `arr[n]` on a list of length `n` (valid indices only go up to `n-1`).
- **Thinking `b = a` copies a list.** It doesn't — both variables reference the same list. Use `.copy()`, `[:]`, or `list(a)` for a real copy.
- **Trying to modify a string in place** (`s[0] = "x"`) — strings are immutable. Convert to a list first if you need character-level edits.
- **Building large strings with repeated `+=` in a loop** — this is quietly O(n²). Prefer collecting pieces in a list and using `"".join()` once.
- **Forgetting slicing's `stop` is exclusive** — `arr[1:4]` gives indices 1, 2, 3, _not_ 4.
