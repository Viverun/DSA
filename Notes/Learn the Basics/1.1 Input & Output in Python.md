### Why I/O Matters in DSA

Before you touch a single algorithm, you need to be fluent in getting data _into_ your program and results _out_ of it. In competitive programming and interview practice, a huge chunk of early bugs come not from broken logic but from mishandled input — wrong types, wrong parsing, off-by-one line reads. Get this rock solid now so it never costs you marks or time later.

---

### 1. Printing Output — `print()`

Python's `print()` function is your primary output tool. No setup, no imports — it's ready to use.

```python
print("Hey, Striver!")
```

**Output:**

```
Hey, Striver!
```

A key behavior to internalize immediately: **`print()` automatically moves to a new line after every call.** You don't have to ask for a newline — it's the default.

```python
print("Hey, Striver!")
print("Hey, Striver!")
```

**Output:**

```
Hey, Striver!
Hey, Striver!
```

---

### 2. Controlling `print()` Behavior

`print()` has optional parameters that control exactly how output is formatted. These matter a lot once you start printing arrays, matrices, or space-separated results (which DSA problems demand constantly).

#### `end` — what to print after the content

By default, `end="\n"`. Override it to change what follows your output:

```python
print("Hey, Striver!", end="")
print("Hey, Striver!")
```

**Output:**

```
Hey, Striver!Hey, Striver!
```

This is the pattern you'll use whenever a problem wants space-separated output on a single line:

```python
for i in range(1, 6):
    print(i, end=" ")
```

**Output:**

```
1 2 3 4 5 
```

#### `sep` — what to print between multiple arguments

`print()` can take multiple comma-separated arguments, and by default joins them with a single space:

```python
print("x", "=", 10)
```

**Output:**

```
x = 10
```

Change the separator with `sep`:

```python
print("x", "y", "z", sep=", ")
```

**Output:**

```
x, y, z
```

#### `flush` — force immediate output

Output is normally buffered for performance. You will almost never need to touch this for DSA problem-solving — it becomes relevant in specific real-world scenarios (live progress logging, mixed output streams). Know it exists; don't worry about it further right now.

### 3. String Formatting — f-strings

Concatenating strings and variables with `+` gets messy fast, especially with non-string types. Python's **f-strings** are the clean, modern, idiomatic way to build formatted output — this is what you should default to.

```python
x = 10
print(f"Value of x: {x}")
```

**Output:**

```
Value of x: 10
```

Multiple variables:

```python
x, y = 10, 20
print(f"Value of x: {x} and y: {y}")
```

**Output:**

```
Value of x: 10 and y: 20
```

You can even do computation inside the braces:

```python
a, b = 4, 5
print(f"Sum is {a + b}")
```

**Output:**

```
Sum is 9
```

**Rule of thumb:** use f-strings for any output that mixes text and variables. Use plain `print(a, b, c)` for quick debugging or simple space-separated values.

### 4. Taking Input — `input()`

`input()` reads a line typed by the user and returns it — **always as a string**, no matter what was actually typed.

```python
x = input()
print(f"Value of x: {x}")
```

**Input:**

```
10
```

**Output:**

```
Value of x: 10
```

This looks fine here, but it's a trap: `x` is currently the _string_ `"10"`, not the _integer_ `10`. If you tried `x + 5` right now, Python would throw a `TypeError` because you can't add an integer to a string. This is the single most common beginner mistake when handling input in Python — **always convert input to the type you actually need.**

---

### 5. Type Conversion on Input

Wrap `input()` with the appropriate type constructor:

```python
x = int(input())
print(f"Value of x: {x}")
```

**Input:**

```
10
```

**Output:**

```
Value of x: 10
```

Now `x` is a real integer, and `x + 5` works correctly.

Common conversions you'll use constantly:

|Function|Converts input to|
|---|---|
|`int(input())`|integer|
|`float(input())`|decimal number|
|`str(input())`|string (redundant — already a string, but explicit)|

### 6. Reading Multiple Values on One Line

A huge number of DSA problems give you input like `10 20` on a single line, expecting you to read both values at once. The standard Python pattern for this:

```python
x, y = map(int, input().split())
print(f"Value of x: {x} and y: {y}")
```

**Input:**

```
10 20
```

**Output:**

```
Value of x: 10 and y: 20
```

**Breaking this down piece by piece — memorize this pattern, you'll type it hundreds of times:**

1. `input()` reads the entire line as one string → `"10 20"`
2. `.split()` breaks it into a list of substrings on whitespace → `["10", "20"]`
3. `map(int, ...)` applies `int()` to every element of that list → produces `10, 20` as integers
4. `x, y = ...` unpacks the two converted values into separate variables

If the line has three values, just unpack three variables: `a, b, c = map(int, input().split())`.

---

### 7. Reading a List of Values (Array Input)

Extremely common pattern: a problem gives you `n` numbers on one line and wants them as a list, not separate variables.

```python
arr = list(map(int, input().split()))
print(arr)
```

**Input:**

```
5 3 8 1 9
```

**Output:**

```
[5, 3, 8, 1, 9]
```

Same logic as before, except instead of unpacking into named variables, you wrap the whole `map` object in `list()` to get a Python list — ready to index, sort, loop over, or pass to any array algorithm.

**Combined pattern** — first read `n` (the count), then read `n` values (very common in problem statements: "first line contains n, second line contains n space-separated integers"):

```python
n = int(input())
arr = list(map(int, input().split()))
print(f"Size: {n}, Array: {arr}")
```

**Input:**

```
5
5 3 8 1 9
```

**Output:**

```
Size: 5, Array: [5, 3, 8, 1, 9]
```

### 8. Reading Multiple Test Cases

Many problems (especially on judges like Codeforces, GFG Practice, LeetCode-style CP sheets) start with a number `t` indicating how many test cases follow, then loop through them:

```python
t = int(input())
for _ in range(t):
    n = int(input())
    arr = list(map(int, input().split()))
    print(f"Test case size {n}: {arr}")
```

**Input:**

```
2
3
1 2 3
4
10 20 30 40
```

**Output:**

```
Test case size 3: [1, 2, 3]
Test case size 4: [10, 20, 30, 40]
```

Note the `_` as the loop variable — this is Python convention for "I don't actually need this value, I just need to loop `t` times."

### 9. Faster Input for Large Inputs (Good Practice to Know Early)

`input()` is fine for almost everything you'll do on this sheet. But if you ever move to a competitive judge with large inputs (thousands of lines), `input()` can be noticeably slow because of the overhead per call. The standard speed-up:

```python
import sys
input = sys.stdin.readline
```

Put this at the top of your file, and every subsequent `input()` call now uses the faster `sys.stdin.readline` under the hood — no other code changes needed. One catch: `readline` keeps the trailing newline character, so when reading plain strings (not doing `.split()` or `int()` conversion) you may need `.strip()`:

```python
import sys
input = sys.stdin.readline

s = input().strip()
```

You don't need this for now — just know it exists for when input size becomes a bottleneck.

|Task|Code|
|---|---|
|Print a value|`print(x)`|
|Print without newline|`print(x, end="")`|
|Print space-separated in a loop|`print(x, end=" ")`|
|Print with custom separator|`print(a, b, sep=", ")`|
|Formatted output with variables|`print(f"x = {x}")`|
|Read a string|`s = input()`|
|Read an integer|`x = int(input())`|
|Read a float|`x = float(input())`|
|Read two integers on one line|`x, y = map(int, input().split())`|
|Read a list of integers on one line|`arr = list(map(int, input().split()))`|
|Read count then array (2-line pattern)|`n = int(input()); arr = list(map(int, input().split()))`|
|Faster input for large data|`import sys; input = sys.stdin.readline`|

### Common Pitfalls to Avoid

- **Forgetting to convert types.** `input()` always gives a string. Doing math or comparisons on unconverted input is the #1 beginner bug here.
- **Mixing up `split()` output with expected count.** If you do `x, y = map(int, input().split())` but the line actually has 3 numbers, Python will raise a `ValueError` for too many values to unpack.
- **Forgetting `list()` around `map()`.** `map()` returns a lazy map object, not a list — it won't show elements when printed directly, and can only be iterated once. Wrap it in `list()` when you need to index into it, sort it, or use it more than once.
